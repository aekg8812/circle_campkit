# アウトドアサークル運営アプリ 開発ガイド（メンバー共有用）

このドキュメントは、これから開発を一緒に進めるメンバー全員に向けた案内です。
プロジェクトの前提・環境構築・作業の進め方をまとめています。**まずこれを読んでから作業を始めてください。**

---

## 1. このアプリは何か

アウトドアサークルの運営（計画立案・参加募集・学校提出用計画書の作成・活動記録）を一元化する Web アプリです。
複数のサークルが使える前提で、ユーザーは「グループ（＝サークル）」に参加または作成し、その中に計画（キャンプ・合宿・日帰り）を作っていきます。

- 技術スタック: **Next.js (App Router) + TypeScript / Supabase / Vercel**
- 詳細な仕様・データモデル・ロール定義は **`docs/DESIGN.md`** にすべて書いてあります。これが設計の「正」です。迷ったら DESIGN.md を見る、が原則です。

## 3. 環境構築（初回だけ）

### 前提ツール

- Node.js（LTS 版）
- Git
- Claude Code
- Supabase のプロジェクトにアクセスできること（下記参照）

### 手順

1. リポジトリを clone する。
   ```
   git clone <リポジトリURL>
   cd <プロジェクト名>
   npm install
   ```
2. `.env.local` を用意する。`.env.local.example` をコピーし、Supabase の接続情報を入れる。

   ```
   cp .env.local.example .env.local
   ```

   必要な値:
   - `NEXT_PUBLIC_SUPABASE_URL`
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY`

   > ⚠️ **`.env.local` は絶対に Git にコミットしない**（`.gitignore` に入っているはず）。
   > 接続情報は Slack の DM など安全な経路で共有し、README やコミットには貼らないこと。

3. 開発サーバーを起動する。
   ```
   npm run dev
   ```
   ブラウザで `http://localhost:3000` を開く。

### Supabase について（重要）

このプロジェクトは **全員で1つの Supabase プロジェクトを共有** します（少人数なので、スキーマとデータの「正」を1箇所に集約するのが一番シンプルなため）。

- すでに Phase 0〜2 の migration は適用済みなので、通常は自分で SQL を流す必要はありません。
- スキーマを変える migration を新しく追加した場合だけ、担当者が **Supabase ダッシュボード → SQL Editor** で流し、その旨をチームに共有します（二重実行を避けるため、流す人は1人に決める）。
- 共有DBなので、**テスト時に他人のデータを消さない**よう注意。動作確認用のグループ名には自分の名前を入れる等のルールにすると安全です。

## 4. Claude Code を使った開発の進め方

このプロジェクトは Claude Code で実装を進めます。全員が同じ流儀で使えるよう、**セッションの最初に下のプロンプトを貼ってから作業を始めてください。**

### セッション開始時に貼る共有プロンプト

```
このリポジトリはアウトドアサークル運営アプリ（Next.js App Router + TypeScript / Supabase / Vercel）です。
作業を始める前に、必ず docs/DESIGN.md を読んでください。

【前提】
- データモデル・ロール・命名規則・RLS 方針は docs/DESIGN.md が「正」。ここに書かれた定義に厳密に従うこと。
- 実装済み: Phase 0（基盤）・Phase 1（アカウントA）・Phase 2（グループ）。

【守ってほしいルール】
1. 1回のセッションで扱うのは「1フェーズ」または「1つの明確なタスク」だけ。指定外のフェーズや機能には手を出さない。
2. DB スキーマを勝手に変えない。変更が必要と判断したら、まず理由を説明して提案し、承認を待つ。
   migration は既存ファイルを書き換えず、supabase/migrations に新しいファイルを追加する。
3. パスワード等の秘密情報を平文で保存・ログ出力しない。認証・権限は必ず RLS / RPC 側で担保する。
4. フォームは zod でバリデーション。更新系は Server Actions または RPC 呼び出しで実装する。
5. 変更後は npm run build が通ることを確認する。
6. 作業完了時に「変更したファイル一覧」「私（人間）が手動で確認すべきこと（Supabase の設定や SQL 実行を含む）」を箇条書きで報告する。

【今回のタスク】
〔ここに作業内容を書く。例: Phase 3（計画 + 行程表 B）を実装してください〕
```

### なぜこのルールなのか（読んでおくと事故が減る）

- **1フェーズずつ**: 一度に大量に生成させるとバグの温床になり、レビューもデバッグも困難になります。小さく作って動作確認、を繰り返します。
- **スキーマを勝手に変えない**: 全員で共有DBを使うので、勝手なスキーマ変更は他の人の環境を壊します。変更は必ず提案 → 合意 → migration 追加 → 共有、の順で。
- **手動確認の報告**: Supabase のバケット作成や SQL 実行など、コードだけでは完結しない作業があります。Claude Code に「人間がやるべきこと」を毎回リストさせると、抜けが減ります。

## 5. コーディング・運用ルール

- **ブランチ運用**: `main` に直接 push しない。フェーズや機能ごとにブランチを切る（例: `feature/phase3-itinerary`）。作業が終わったら Pull Request を出し、他の人が1人以上レビューしてから main にマージ。
- **1PR = 1まとまり**: フェーズ全部を1PRにせず、レビューしやすい単位に分ける。
- **DESIGN.md が正**: 仕様の疑問は DESIGN.md を確認。書いていない・迷う場合はチームで相談してから DESIGN.md を更新し、それから実装する（実装が先行して仕様がズレるのを防ぐ）。
- **migration は追記のみ**: 既存の migration ファイルは編集しない。変更は新しいファイルで。ファイル名は日時プレフィックス（例: `20240301000000_phase3_plans.sql`）。
- **秘密情報**: `.env.local`・Supabase のキー・グループのパスワードなどをコミットや公開チャンネルに貼らない。

## 6. 残りのフェーズと担当の決め方

Phase 3 以降は、フェーズ単位で担当を割り振ると並行しやすいです（ただしフェーズ間に依存があるので順序に注意）。

- **Phase 3（計画 + 行程表 B）**: 他フェーズの土台になるので最優先。
- **Phase 4（募集 C）**: Phase 3 の plans が前提。
- **Phase 5（計画書 D）**: Phase 3・4 のデータが揃ってから。PDF 出力（日本語フォント埋め込み）があるので少し重い。
- **Phase 6（準備 E + アーカイブ F）**: 比較的独立していて着手しやすい。

各フェーズの詳細な実装内容は `docs/DESIGN.md` に定義済みです。担当者は DESIGN.md の該当フェーズを読み、上の「共有プロンプト」の【今回のタスク】に落とし込んで Claude Code に依頼してください。

## 7. 困ったとき・確認コマンド

- ビルドが通るか: `npm run build`
- DB の中身を直接確認したいとき（Supabase → SQL Editor）:
  ```sql
  select * from group_members;   -- 誰がどのグループに何の役職で入っているか
  select * from plans;           -- 作成された計画
  ```
- 表示とDBの実態がズレていたら、まず SQL Editor で該当テーブルを直接見て切り分ける。
- RLS で「読めない・書けない」系のエラーは、DESIGN.md の RLS 方針とポリシーを確認する。
